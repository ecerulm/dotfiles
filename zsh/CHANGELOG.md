## [2026-05-28]

### Docs

- `AGENTS.md` (`CLAUDE.md` symlink): new "awk on macOS: BWK gotchas" section documenting two patterns that silently misbehave under BWK awk (the macOS default) and have already cost real debug time in this repo. (1) `awk '...' fileA -` reads `-` as stdin only when fileA is non-empty; with a zero-byte first file BWK silently drops the stdin stream, leaving the second-file body unreached and the pipeline succeeding with empty output. (2) `awk -v var=$(command_with_newlines)` fails with `awk: newline in string ...` because BWK rejects embedded newlines in `-v` values. Section covers the wrong/right patterns (always `mktemp` real files for awk's inputs; gate two-file awk passes on `[[ -s $first_file ]]`), explains why `/dev/stdin` is not a fix, and ends with a "symptom card" — "awk-using pipeline silently produces zero output despite the upstream stages looking healthy" → first suspects are `-` after an empty mktemp file, and `-v var=$(emits-newlines)`. Inserted between the existing "Emitting Escape Sequences from zsh" and "fzf Conventions" sections.

### Fixed

- `rlm-dbt-find-source-code`: the MRU sort awk pass produced zero output, so the picker was always empty (symptom: select `--- REFRESH CACHE ---` and the next picker has no entries — even though `~/.cache/rlm-dbt/<key>/nodes.json` has thousands of nodes). Two macOS-BWK-awk gotchas conspired: (1) the original layout `awk '...' "$_hist_tmp" -` reads `-` as stdin only when the preceding file `$_hist_tmp` is **non-empty** — when history was empty (zero bytes) BWK awk silently skipped the `-`, leaving the entries stream unread and `tail_count` empty, so nothing reached the picker; (2) the alternative `-v hist=$(...)` approach to pass history as a string fails with `awk: newline in string ...` because BWK awk rejects embedded newlines in `-v` values. Fix: gate the whole MRU pass on `[[ -s $_hist_keys_tmp ]]` — when there's no history, skip awk entirely and use the entries straight (already sort-uniq'd by the upstream `jq | sort -u` pipeline). When history *does* exist, write the entries to a second temp file so awk reads two real files (no `-` stdin), eliminating the BWK quirk. Also dropped the two O(n²) bubble sorts that fell out of the original awk (entries are sort-uniq'd upstream; history is bounded by the shared TSV cap at 500 lines).

### Added

- `rlm-dbt-find-source-code` (short alias `dbt-find-source-code`): new autoloaded function. fzf picker over every resource in the current dbt project (model, seed, test, snapshot, source, analysis, exposure). Entries render as `<resource_type>:<short_name>` (e.g. `source:storytel_order_service.state`, `model:dim_customer_demographics_latest`) so typing `source:` or `model:` narrows by type — the short_name is the dbt `unique_id` with the leading `<resource_type>.<package>.` stripped via `jq … sub("^[^.]+\\.[^.]+\\."; "")`. Reuses every project-discovery / venv-bootstrapping step from `rlm-dbt-ls` (`_rlm-python-ensure-version`, `_rlm-poetry-ensure-venv`, `_rlm-dbt-ensure-deps`, `_rlm-dbt-project-key`, `_rlm-dbt-nodes-cache`, `_rlm-dbt-history`) and reads from the shared per-project nodes cache at `~/.cache/rlm-dbt/<project_key>/nodes.json` — no per-call `dbt ls`. MRU-sorted via the shared pooled history under the command tag `dbt-find-source-code`; the MRU rank logic lives in an awk two-pass over (history-temp-file, entries-stdin) using the line number in the history file as the rank (file-based rather than `-v hist=…` because awk's `-v` rejects embedded newlines in the value, with a misleading "newline in string" error). First picker entry is `--- REFRESH CACHE (cached <age>) ---` so a stale cache can be busted in-picker without losing history. Preview pane renders the resource's source file via `bat --style=plain --color=always` when installed, falling back to `cat`. On confirmation: prints the absolute file path to stdout, copies it to the system clipboard (pbcopy/xclip/xsel), and opens `$EDITOR` on the file (`"${=EDITOR}"` to honor compound editor commands like `code --wait`). Requires `$EDITOR` set; aborts with a diagnostic otherwise. Follows the standard `rlm-*` autoload pattern: `autoload -Uz` line in `.zshrc` (also reordered the existing dbt autoloads/aliases alphabetically while there), short alias, real `helpdir/rlm-dbt-find-source-code` + symlink `helpdir/dbt-find-source-code`, README entry.

### Changed

- `_rlm-dbt-nodes-cache`: extend the shared per-project nodes cache to cover every dbt `resource_type`. The `refresh` subcommand's `dbt ls --output json` invocation now also includes `--resource-type source --resource-type analysis --resource-type exposure` alongside the existing `model + seed + test + snapshot` set, so the on-disk `~/.cache/rlm-dbt/<project_key>/nodes.json` is the complete project catalog. AGENTS.md (`CLAUDE.md` symlink) updated to mention the broader coverage and the new caller.

### Changed

- `rlm-git-squash-branch`: harden the workflow and prepopulate the commit message. New preconditions / steps, in order: (1) verify the working tree is clean via `git status --porcelain` — if dirty, prints `git status --short` to stderr and aborts; squashing on top of uncommitted changes would entangle them with the squash commit and a failed `pull --rebase` would leave the user in a worse state than they started. (2) `git pull --rebase` to sync any remote commits onto the local branch. (3) `git push` to durably persist the pre-squash commits to the remote, so the original history is recoverable from the remote reflog if the squash or subsequent force-push goes wrong. (4) The existing `git reset --soft <merge_base>` and `git commit -s`, now upgraded to `git commit -s -e -F -` with the messages of all squashed commits piped in as the initial editor buffer — header `Squashed commit of <branch>:`, blank line, then `git log --reverse --no-merges --format='* %s%n%n%b' <merge_base>..HEAD` (one `* <subject>` + body block per commit, oldest first — the same shape `git merge --squash` produces). `-F -` reads the message from stdin, `-e` forces the editor to open with that message as the initial buffer so the user can edit, accept, or abort by emptying it. (5) `git rebase <default_branch>` so the squashed commit sits on the current default-branch tip — the `git pull --rebase` above only synced commits on the current branch; the default branch on origin may have moved on independently, and rebasing now means the eventual force-push lands cleanly without the reviewer having to remember to update the branch separately. `helpdir/rlm-git-squash-branch` and the README row updated to describe the new sequence.

### Changed

- `rlm-pr-worktree`: add an in-picker way to refresh the shared `~/.cache/rlm-jira/` cache, so callers no longer have to bounce out to `rlm-jira-pick` (or hand-invoke `_rlm-jira-cache refresh`) when the JIRA rows have gone stale. Two affordances, picker-equivalent: (1) a `--- REFRESH JIRA CACHE (cached <age>) ---` sentinel emitted as the very first input line whenever acli is reachable (so it always lands at the top of the picker — fzf's default ordering preserves input order absent a query), with the age suffix served from `_rlm-jira-cache age` and rendered as `unknown` when the cache file does not yet exist; (2) a `Ctrl-R` keybinding wired up via `--expect=ctrl-r`, working from any row. Both paths call `_rlm-jira-cache refresh` and then recursively re-invoke `rlm-pr-worktree`, so the user sees a fresh picker without having to re-run the command. The `--expect=ctrl-r` machinery requires the selection-readback to peel a leading key line off fzf's stdout — handled with a `${selection%%$'\n'*}` / `${selection#*$'\n'}` split, with a defensive fallback when the second segment equals the first (the no-newline case fzf should never produce but worth guarding). The REFRESH branch is gated on `jira_available` so Ctrl-R prints a "JIRA unavailable" diagnostic instead of silently no-oping when acli is not installed. Preview pane gets a `REFRESH` arm that documents both the sentinel and the keybinding. Header line gains the `Ctrl-R: refresh JIRA cache` hint. Top-of-file comment block and `helpdir/rlm-pr-worktree` updated; the pre-existing "to force a refresh, run rlm-jira-pick" hint is rewritten to mention the new in-picker affordances first.

### Added

- `rlm-pr-worktree-rm-merged-closed` (short alias `pr-worktree-rm-merged-closed`): new autoloaded function. Automatically deletes every git worktree in the current repo whose branch has a GitHub PR that is already merged or closed. Builds the same branch → PR-state map as `rlm-pr-worktree-rm` (one `gh pr list --state all --limit 500 --json number,state,headRefName` call), walks `git worktree list --porcelain`, and selects worktrees whose branch resolves to a PR in state `merged` or `closed`. The current worktree and the main/primary worktree are always excluded (same gate as `rlm-pr-worktree-rm`); worktrees with `no-pr` or `open` PR state are left alone — this function only acts on PRs in a terminal state. Lists the candidates column-aligned (`<rel-path>  <branch>  <state>#<N>`) and prompts y/N before deleting; `--yes` / `-y` skips the prompt for cron/scripted use, `--dry-run` / `-n` prints the list and exits 0 without deleting. Each removal is `git worktree remove`, retried with `--force` on failure (matches `rlm-pr-worktree-rm` semantics — uncommitted changes in a worktree whose PR has already landed are nearly always stale rebase/merge debris). `git worktree prune` runs once at the end. Follows the standard `rlm-*` autoload pattern: `autoload -Uz` line in `.zshrc`, short alias, real `helpdir/rlm-pr-worktree-rm-merged-closed` + symlink `helpdir/pr-worktree-rm-merged-closed`, README entry.

### Fixed

- `rlm-pr-worktree`: when creating a worktree for a PR, run `gh pr checkout <N> -f` inside the new worktree after `git worktree add` to wire up upstream tracking. Previously the function did `git fetch origin pull/N/head:pr-N` + `git worktree add <path> <pr_branch>`, which created a local branch from the fetched ref but never set `branch.<name>.remote` / `branch.<name>.merge`, so the subsequent `git pull` failed with "There is no tracking information for the current branch." `gh pr checkout` handles same-repo PRs (`--track origin/<branch>`) and fork PRs (sets `branch.<name>.remote` to the fork URL and `branch.<name>.merge` to `refs/heads/<headRef>`) identically to what `gh pr checkout` does in a non-worktree checkout. The `-f` flag makes it idempotent when re-entering an existing worktree (resets the local branch to the PR head). A `gh pr checkout` failure prints a warning but does not abort — the worktree still exists and is usable, just without tracking.

### Changed

- `rlm-afw-deploy`: can now be invoked from anywhere inside the project — the function walks up from `$PWD` looking for the first directory that contains both `airflow_manager/` and `pyproject.toml` (typically `.../airflow/` in the orchestration repo) and `cd`s there before proceeding. Previously the two precondition checks (`[[ -d airflow_manager ]]` and `[[ -f pyproject.toml ]]`) demanded `$PWD == project_root` and aborted with "run from project root" otherwise, forcing the caller to remember the layout. The walk-up replaces both checks with a single loop that climbs `${dir:h}` until it finds the marker pair or hits `/`; if no ancestor matches, the new error message names *both* markers so the failure mode is self-describing instead of "run from project root" (which gave no hint about which dir to cd to). When the walk-up succeeds in a non-root cwd, prints `afw-deploy: cd <project_root>` to stderr before the `cd` so the user sees where they landed (a one-line announcement matching the style of the other diagnostic prints in the function). Every downstream step (`_rlm-poetry-ensure-venv`, the `poetry run afw` invocation, the fzf picker preview) already operates relative to `$PWD`, so retargeting the cwd up-front is sufficient — no need to thread `$project_root` through individual command invocations. `helpdir/rlm-afw-deploy` updated: synopsis paragraph now says "can be invoked from anywhere inside the project" and step 1 of the "WHAT IT DOES" list describes the walk-up + cd.

## [2026-05-25]

### Fixed

- `rlm-pr-worktree`: when invoked from inside a secondary git worktree, place the new worktree as a sibling of the **main** repo (and default its name from the main repo's basename) instead of as a sibling of whichever worktree happened to be the cwd. Previously, calling `pr-worktree` from inside `~/code/myrepo-pr-42/` proposed `~/code/myrepo-pr-42-pr-99/` (sibling of the worktree, basename of the worktree) instead of `~/code/myrepo-pr-99/` (sibling of the main repo, basename of the main repo). Root cause: `repo_root` was derived from `git rev-parse --show-toplevel`, which returns the **current worktree's** root, not the main repo's root. Fix: added a parallel `main_root` derived from `git rev-parse --path-format=absolute --git-common-dir` (whose `:h` strips `.git` and lands on the main worktree regardless of the cwd), use `${main_root:h}` for the new worktree's parent path in all three `_prwt_make_*_worktree` helpers + the `_prwt_prompt_dirname` vared prompt, and use `${main_root:t}` for `repo_name` (the basename used in default dirnames like `<repo>-pr-<N>` / `<repo>-<KEY>`). All `git -C "$repo_root"` operations are unchanged — `git worktree add`, `fetch`, `show-ref` etc. all operate on the shared common-dir regardless of which worktree is the cwd, so there's no need to retarget them at the main worktree. Falls back to `repo_root` for `main_root` if `--git-common-dir` ever returns empty (defense-in-depth; should not happen for a valid repo).

### Added

- `rlm-git-find-in-remotes` (short alias `git-find-in-remotes`): new autoloaded function. Given an extended regex (and an optional refname glob, default `refs/remotes/`), iterates every matching remote-tracking branch via `git for-each-ref`, lists its tracked paths with `git ls-tree -r --name-only "$branch"`, and pipes through `grep -E -- "$pattern"`. Symbolic refs such as `refs/remotes/origin/HEAD` are skipped (via `git symbolic-ref --quiet`) so each commit is reported once. Output is one block per matching branch: the branch name (with the `refs/remotes/` prefix stripped) followed by each matching path indented two spaces. Use case: locate a file that exists on some remote branch but not your current checkout (e.g. "which branch added `path/to/foo.sql`?") without needing `git log --all -- <path>` or to clone each branch. Exits 0 when at least one match was found, non-zero otherwise. The optional second arg restricts scope (e.g. `refs/remotes/origin/` for a single remote, `refs/remotes/origin/feat/` for a branch-name prefix). Requires the working remote-tracking refs to be up to date — run `git fetch --all` first if you suspect stale state. Follows the standard `rlm-*` autoload pattern: `autoload -Uz` line in `.zshrc`, short alias, real `helpdir/rlm-git-find-in-remotes` + symlink `helpdir/git-find-in-remotes`, README entry.

- `_rlm-dbt-cmd`: after the fzf picker selects a node, open the selector in a `vared` one-line editor so the user can tack on graph operators (`my_model+`, `+my_model`, `my_model+1`) or switch to a tag/path selector before dbt runs. Enter accepts the picked value as-is; Ctrl-C aborts with `return 130` to match the existing fzf abort convention; an empty selector after edit is rejected with `return 1`. Only fires when the picker was used (i.e. `-s` was not passed on the command line), so `dbt-run -s foo` still skips straight to the cost estimate. Affects all three wrappers (`rlm-dbt-run` / `rlm-dbt-build` / `rlm-dbt-test`) since the edit happens in the shared helper. `helpdir/rlm-dbt-{run,build,test}` updated to document the new behavior.

- `_rlm-python-ensure-version`: new internal helper. Given `(cwd, label)` it runs `python -EsSc 'import sys; print(sys.executable)'` from `cwd` — the same probe Poetry uses internally when bootstrapping a venv. On non-zero exit (pyenv's shim emits a "no such version" message and returns 127 when there's no resolvable `.python-version` in the ancestry) prints a multi-line diagnostic (failing command, cwd, captured output, hint about `.python-version` / `pyenv install`) and returns 1. Catches the failure mode that previously surfaced several layers down as a misleading "poetry install failed" — poetry shells out to `python -EsSc ...` to discover the interpreter, and pyenv's helpful "no such version" message gets buried in `subprocess.CalledProcessError`. The cwd matters because pyenv only walks up from the cwd to find `.python-version` — in layouts where `pyproject.toml` lives in a parent dir and `.python-version` lives in a child (the work repo's `data-platform-dbt-.../storytel-dbt/.python-version` is the motivating case), callers must pass the child dir, not the pyproject.toml parent. Follows the established `_rlm-*` convention (no autoload line in `.zshrc`, no short alias, no `helpdir/` entry; callers `autoload -Uz` at the top of their bodies).

- `rlm-dbt` (short alias `dbt`): generic dbt dispatcher. Behavior: (1) when called as `rlm-dbt <subcmd> [args...]` and `rlm-dbt-<subcmd>` exists as a function or alias, delegate to that wrapper passing the remaining args through — so `dbt run` opens the `rlm-dbt-run` fzf picker, `dbt run -s tag:finance` forwards `-s tag:finance` to `rlm-dbt-run`, `dbt ls`/`dbt build`/`dbt test`/`dbt sandbox` likewise hit their existing wrappers; (2) otherwise invoke plain dbt via `_rlm-dbt-bin` (so it respects `DBT_USE_FUSION` just like the rest of the `rlm-dbt-*` family). For the plain-dbt path, auto-injects `--project-dir <root>` by walking up from `$PWD` looking for the nearest `dbt_project.yml`, and `--profiles-dir "$DBT_PROFILES_DIR"` when that env var is set — both injections are skipped when the user has already supplied the flag in the args (detected via the zsh `${@[(I)--project-dir]}` reverse-index trick, so a single pass over the arg list catches both `--project-dir=foo` and `--project-dir foo` forms without needing a separate parser). Wrapper-detection uses `(( ${+functions[$wrapper]} ))` and `(( ${+aliases[$wrapper]} ))` so the function works whether the wrapper is autoloaded-and-already-resolved, autoloaded-but-not-yet-called, or just an alias. Short alias `dbt='rlm-dbt'` shadows the system `dbt` binary — safe because `_rlm-dbt-bin` never calls bare `dbt` (Fusion mode resolves `${aliases[dbtf]}` to an absolute path; non-Fusion mode uses `poetry run dbt`), so the shadowing only affects what the user types interactively, not what runs underneath. Use cases: a single muscle-memory entry point (`dbt parse`, `dbt deps`, `dbt debug`, `dbt source freshness`) without losing the picker behavior of the existing `rlm-dbt-{run,build,test,ls}` wrappers. Help via `dbt --help`. Plain-dbt invocations echo the full resolved command line to stderr before execution (prefixed `+ ` to match `set -x` convention) so the user can see exactly which dbt binary is being invoked and which `--project-dir` / `--profiles-dir` / etc. flags were auto-injected. Args containing whitespace / globs / shell-special chars are single-quoted via zsh's `${(q-)x}` parameter expansion so the echoed line is paste-safe. The echo lands on stderr so `$(rlm-dbt parse)` and other command substitutions still capture only dbt's stdout. Wrapper-delegation paths do not echo here — the wrappers (`rlm-dbt-run` etc.) own their own command construction and should print their own commands if they want to.

- `_rlm-jira-cache`: per-group fetch cap lowered from 1500 to 70. Each of the four JQL queries returns at most 70 rows, so the upper bound on the cache is 4 × 70 = 280 rows before cross-group dedupe (down from a theoretical 6000). Two reasons: (1) the picker stays snappy and visually scannable — each group renders as roughly one screenful of fzf candidates rather than a multi-thousand-row scroll dominated by the DATA-only bucket; (2) the upstream `{ ... } | column -t | fzf` pipe in `rlm-pr-worktree` never gets large enough to risk the SIGPIPE-into-pipefail silent-exit class of bugs (see the corresponding `### Fixed` entry below). The active working set on each group is small in practice — assigned/reporter/watching are bounded by what one person personally touches, and the DATA-only bucket is already ordered by `updated DESC`, so 70 covers everything that's been touched in the past few weeks. Older tickets remain findable via the JIRA web UI or by typing the KEY directly into `rlm-pr-worktree` / `rlm-jira-open`.

- `_rlm-jira-cache`, `_rlm-jira-history`, `_rlm-jira-sorted-rows`: picker is now sorted by a single composite key — `max(history_picked_at, updated_epoch, last_commented_epoch)` descending — instead of the old "history rows first, then the rest" interleave. A freshly-commented ticket can now outrank an old picked one; a never-picked-but-recently-updated ticket can outrank a picked-but-stale one; the 4-group priority survives only as a tiebreaker for the (extremely rare) case where two rows share a sort key. (b) `history.txt` format becomes TSV `<epoch>\t<KEY>` so the sort can read `picked_at` per KEY; the legacy single-column format auto-upgrades on first read/append/migrate-legacy invocation via an in-place awk rewrite that assigns synthetic timestamps `now - rank * 60` (rank=0 for the bottom/newest line), preserving the existing MRU order exactly — atomic `mv` from a temp file, idempotent (a no-op if any line in the file already contains a tab). The `migrate-legacy` subcommand also runs the format-upgrade when copying from `~/.cache/jira-pick/history.txt`. (c) `_rlm-jira-sorted-rows` replaces its hand-rolled "history-first then cache" interleave with an inline Python composite-key sort that reads cache + history, builds `picked_at[KEY] = max(epoch)`, computes `sort_key = max(picked_at.get(key, 0), updated_epoch, last_commented_epoch)` per cache row, and emits in `(sort_key DESC, group_idx ASC)` order. (d) Per-group cap drops from 70 to 20 (so 4 × 20 = 80 raw rows before dedupe), which keeps phase B's per-issue fetch bounded at ~8s. (e) Both callers' parsers bump from `len(parts) < 5` / `NF>=5` to `< 7` / `NF>=7` to reflect the new row width; the renderers still use only fields 1–4 (display) and 1–5 (pr-worktree reshape), so the visible picker columns are unchanged.

### Fixed

- `rlm-pr-worktree`, `rlm-jira-pick`: silent exit immediately after picking an entry. Symptom: the picker rendered correctly, the user picked a row, control returned to the prompt — no vared prompt, no worktree, no error message. Root cause: the picker subshell `selection=$(... | column -t | fzf)` ran under the function-scope `set -o pipefail`. fzf stops reading from stdin once it has enough rows to render the first screenful; upstream `column -t` (and, in `rlm-jira-pick`, `_rlm-jira-sorted-rows`'s `cat` plus the inline `python3` renderer) then write into a paused pipe, eventually get SIGPIPE and exit 141. `pipefail` propagates that 141 out through the `$(...)` substitution, and the `|| return 130` immediately after the close-paren fires — silently exiting the function right after the user picked, before any vared prompt or worktree-creation logic could run. The bug was latent before the `_rlm-jira-cache` refactor because the old per-call `acli jira workitem search --limit 100` produced too few rows to trigger SIGPIPE in practice; the shared cache (even capped at 70 per group) is wide enough that the SIGPIPE actually fires. Fix: wrap each picker subshell with `setopt local_options no_pipefail` so pipefail is disabled only inside the subshell — the outer pipefail, which legitimately guards the jq pipelines that build `pr_rows` / `other_pr_rows` / `jira_rows` upstream in `rlm-pr-worktree`, stays in effect. A multi-paragraph header comment above the affected `selection=$(` in each file documents the SIGPIPE chain so a future edit doesn't reintroduce the bug.

Every section is independently best-effort: missing JIRA key, missing PR, missing `gh`/`acli`/`fd` all just omit the corresponding line rather than failing the preview.
